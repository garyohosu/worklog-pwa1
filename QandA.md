### Q1. 自動進行ループの主制御主体はどれか（セクション4・13）

Claude Code・Codex CLI・回答担当AIの3者が登場するが、セクション13の自動進行ループをどの主体が駆動するのかが明示されていない。
Claude Code が主制御を担当するという理解でよいか。

**A1. Claude Code を主制御主体とする。**

WORKFLOW.md の役割分担では、Claude Code は「作成・改訂・進行管理・次工程への進行制御・state.json の初期生成」を担当している。一方、Codex CLI はレビュー担当、回答担当AIは質問回答担当であり、全体ループの駆動主体として最も自然なのは Claude Code である。したがって、セクション13の自動進行ループは Claude Code が state.json を読み、必要に応じて Codex CLI や回答担当AI を呼び出して進行するものと定義する。

### Q2. 再試行回数の上限はあるか（セクション19.1）

一時的エラー時は「再試行対象」とあるが、上限回数が未定義のため無限ループに陥るリスクがある。
上限回数を設けるか、またはその値はいくつにするか。

**A2. 一時的エラーの自動再試行回数は 3 回を上限とする。**

セクション19.1では一時的エラーを再試行対象としているが、上限がないと無限ループの危険がある。初期運用では単純で扱いやすい固定値が向いているため、3回を標準値とするのが妥当である。
3回失敗した場合は `blocking_issues` に記録し、対象成果物の状態を `blocked` とし、人間または回答担当AIの判断待ちへ移行する。

### Q3. 回答ファイルが部分的に存在する場合の扱い（セクション13 ステップ7）

「質問ファイルがあり、回答ファイルがなければ回答生成する」とあるが、回答ファイルが部分的にしか存在しない場合（質問の一部のみ回答済み）はどう扱うか。
再生成するか、差分のみ追記するか。

**A3. 差分追記を基本とし、未回答項目のみ追加生成する。**

回答ファイルが部分的に存在する状態は実運用で起こりやすい。これを毎回全再生成にすると、既存の判断や文章が上書きされ、履歴追跡もしにくくなる。
そのため、回答ファイルが部分的に存在する場合は、既存回答を保持し、未回答の質問だけを追加生成する方式とする。
ただし、質問文自体が変更された場合や回答整合性が崩れた場合は、対象回答ファイル全体を再生成してよい。

### Q4. `IMPLEMENTATION_PLAN.md` を承認対象から省略できる条件（セクション16）

「必要に応じて承認対象に含める」とあるが、含めない場合の判断基準が不明。
どのような場合に省略可能か。

**A4. 実装方針が SPEC.md・CLASS.md・TEST.md に十分具体化されており、別途実装計画書を作る必要がない小規模案件では省略可能とする。**

省略可能なのは以下のようなケースである。

- 小規模または単機能である
- 実装単位が明確で、タスク分解が不要
- 実装順序や分割方針が設計書から十分読み取れる
- 外部連携や移行作業など、追加の実装計画が不要

逆に、複数モジュール・複数画面・複数API・移行手順・段階リリースなどがある場合は、`IMPLEMENTATION_PLAN.md` を承認対象に含める。

### Q5. `review_requested` 状態の遷移タイミング（セクション7・13）

状態定義に `review_requested` が存在するが、セクション13の自動進行ループにはこの状態への遷移が記述されていない。
`review_requested` はどのタイミングで付与され、誰が付与するか。

**A5. `review_requested` は、成果物生成または改訂が完了し、レビュー担当AIへレビュー依頼を出した時点で Claude Code が付与する。**

自然な状態遷移は次の通り。

- 新規作成直後: `drafted`
- Claude Code がレビュー依頼を発行: `review_requested`
- レビュー報告生成後: `reviewed`
- 質問あり: `questions_pending`
- 回答反映後: `revised`
- 再レビュー依頼時: 再度 `review_requested`
- 承認時: `approved`

したがって、`review_requested` は Claude Code が外部レビュー処理を起動した時点で設定する状態と定義する。

### Q7. `WORKFLOW.md` 単体起動時に補助ディレクトリも自動生成するか（セクション10・14・20）

最小構成を `WORKFLOW.md` 1ファイルとする一方で、推奨構成では `state/` `reviews/` `qa/` `logs/` `prompts/` が前提になっている。
ブートストラップ時にこれらのディレクトリや初期ファイルも自動生成する前提か。それとも `state.json` だけを生成し、他は遅延生成する前提か。

**A7. ブートストラップ時に最低限必要な補助ディレクトリは自動生成する。**

`WORKFLOW.md` 単体起動を成立させるため、起動時に以下を自動生成する。

- `state/`
- `reviews/`
- `qa/`
- `logs/`

一方、`prompts/` は固定テンプレート運用が未確定のため初回ブートストラップでは必須とせず、必要時に遅延生成する。
これにより、「`WORKFLOW.md` だけで起動できる」という説明と推奨ディレクトリ構成を両立できる。

### Q8. `IMPLEMENTATION_PLAN.md` を不要とする場合の `artifact_order` と完了判定の扱い（セクション6・15・16・21）

セクション16では `IMPLEMENTATION_PLAN.md` を「必要に応じて承認対象に含める」としているが、セクション15のテンプレートでは常に `artifact_order` と `artifacts` に含まれており、セクション21では「すべての成果物が approved」を完了条件としている。
不要案件では `IMPLEMENTATION_PLAN.md` を state から除外するのか、あるいは `skipped` のような別状態を導入するのか。

**A8. `IMPLEMENTATION_PLAN.md` は任意成果物として扱い、不要案件ではブートストラップ時に `artifact_order` と `artifacts` から除外する。完了条件は「必須成果物が approved」に変更する。**

必須成果物は `SPEC.md`, `USECASE.md`, `SEQUENCE.md`, `CLASS.md`, `TEST.md` とする。  
`IMPLEMENTATION_PLAN.md` は任意成果物であり、必要な案件でのみ有効化する。

ブートストラップ時に `IMPLEMENTATION_PLAN.md` が不要なら `artifact_order` と `artifacts` に含めない。  
完了条件は「必須成果物が approved、かつ有効化された任意成果物があればそれも approved」とする。

これにより、`skipped` のような追加状態を導入せずに、設計文書と状態モデルを整合させられる。

### Q10. 成果物の改訂による後続成果物への影響（再検証）の制御

`SPEC.md` などが承認後に改訂（`revised`）された場合、既に承認済みの後続成果物（`USECASE.md` など）の承認状態をどう扱うか。自動的に `review_requested` に戻すべきか、あるいは影響範囲を人間が判断するのか。

**A10. 先行成果物が承認後に改訂された場合、依存する後続成果物は自動的に「再検証待ち」に戻す。具体的には approved を外し、review_requested に遷移させる。**

*   **Reason:** このワークフローは「承認されるまで次工程へ進まない」「改訂後は必ず再レビューする」が基本原則なので、上流成果物の変更後に下流成果物の承認をそのまま維持すると、設計整合性が崩れる。したがって、`SPEC.md` や `USECASE.md` のような先行成果物が `revised` になった時点で、それに依存する後続成果物は再検証対象とする。
*   **影響範囲:** 依存関係に基づく「限定的な伝播」とし、全成果物を一律に戻すのではなく、たとえば `SPEC.md` 改訂なら `USECASE.md` 以降、`CLASS.md` 改訂なら `TEST.md` 以降だけを戻す。これで過剰な差し戻しを避けられる。

### Q11. 実装フェーズにおける「実装レビュー」の具体的な主体と基準

セクション17で「実装レビューが `APPROVED`」であることが条件とされているが、このレビューを行うのは Codex CLI か。また、ソースコードのレビューにおいて、設計文書との整合性をどう担保するか（プロンプトや入力ファイルの指定方法）。

**A11. 実装レビューの主体は Codex CLI とし、入力には実装コードに加えて承認済みの設計文書一式を与える。**

*   **Reason:** `WORKFLOW.md` では Codex CLI はレビュー担当として定義されているため、実装レビューもその延長で扱うのが自然である。
*   **レビュー基準:** 少なくとも次の4点とする。
    1.  `SPEC.md` との要件整合
    2.  `USECASE.md` / `SEQUENCE.md` / `CLASS.md` との設計整合
    3.  `TEST.md` の想定を満たす実装か
    4.  明らかな欠陥、未使用コード、責務逸脱がないか
*   **運用:** Codex CLI に対して 実装コード＋承認済み設計文書＋レビュー観点をまとめて渡す。つまり、実装レビューは単なるコードレビューではなく、設計準拠レビュー と位置付ける。

### Q12. `state.json` 内の `blocking_issues` と `human_decisions_pending` のデータ構造

テンプレートでは空配列だが、実際に記録する際のデータ構造（例：`id`, `description`, `artifact`, `timestamp` など）が定義されていない。機械的な処理（通知やフィルタリング）のために標準化が必要ではないか。

**A12. 両方ともオブジェクト配列とし、最低限 id, artifact, description, created_at, status を持つ標準構造にする。**

*   **Reason:** 空配列のままだと機械処理しにくいので、通知・再開・絞り込みのために構造を固定したほうがよい。標準形は次のようにする。
    ```json
    {
      "id": "BI-001",
      "artifact": "SPEC.md",
      "description": "認証方式の選択が未確定",
      "created_at": "2026-03-24T14:30:00+09:00",
      "status": "open"
    }
    ```
    blocking_issues には必要に応じて severity, source, retry_count を追加してよい。human_decisions_pending には options や recommended_option を追加して、回答担当AIや人間が判断しやすくする。これで state/state.json の「機械可読に管理する」という原則と整合する。

### Q13. 自動進行ループの終了条件とタイムアウト

セクション13のループにおいて、`done` 以外でループを抜ける条件（`blocked` 継続時など）や、1サイクルあたりの最大実行時間、あるいは最大サイクル数などの制約はあるか。

**A13. ループは done 以外にも、blocked 継続、最大サイクル数超過、最大実行時間超過で停止する。**

*   **Reason:** 現状の文書では done 到達時の完了はあるが、異常系のループ脱出条件が弱い。実運用では次を定義しておくと安定する。
    *   **1サイクルの最大実行時間:** 15分
    *   **連続実行の最大サイクル数:** 20サイクル
    *   **同一成果物で blocked が継続した場合:** 即停止
    *   **同一エラーの再試行上限:** A2 の通り 3 回
*   **停止時の挙動:** いずれかに達した場合は final_status を paused または blocked に更新し、詳細をログに出力してループを抜ける。

**A14. prompts/ 内にファイルが存在する場合、それを最優先のカスタムプロンプトとして使用する。**
