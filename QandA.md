# QandA.md

## WORKFLOW.md レビュー時の不明点

---

### Q1. 自動進行ループの主制御主体はどれか（セクション4・13）

Claude Code・Codex CLI・回答担当AIの3者が登場するが、セクション13の自動進行ループをどの主体が駆動するのかが明示されていない。
Claude Code が主制御を担当するという理解でよいか。

**A1. Claude Code を主制御主体とする。**

WORKFLOW.md の役割分担では、Claude Code は「作成・改訂・進行管理・次工程への進行制御・state.json の初期生成」を担当している。一方、Codex CLI はレビュー担当、回答担当AIは質問回答担当であり、全体ループの駆動主体として最も自然なのは Claude Code である。したがって、セクション13の自動進行ループは Claude Code が state.json を読み、必要に応じて Codex CLI や回答担当AI を呼び出して進行するものと定義する。

### Q2. 再試行回数の上限はあるか（セクション19.1）

一時的エラー時は「再試行対象」とあるが、上限回数が未定義のため無限ループに陥るリスクがある。
上限回数を設けるか、またはその値はいくつにするか。

**A2. 一時的エラーの自動再試行回数は 3 回を上限とする。**

セクション19.1では一時的エラーを再試行対象としているが、上限がないと無限ループの危険がある。初期運用では単純で扱いやすい固定値が向いているため、3回を標準値とするのが妥当である。
3回失敗した場合は `blocking_issues` に記録し、対象成果物の状態を `blocked` とし、人間または回答担当AIの判断待ちへ移行する。

### Q3. 回答ファイルが部分的に存在する場合の扱い（セクション13 ステップ7）

「質問ファイルがあり、回答ファイルがなければ回答生成する」とあるが、回答ファイルが部分的にしか存在しない場合（質問の一部のみ回答済み）はどう扱うか。
再生成するか、差分のみ追記するか。

**A3. 差分追記を基本とし、未回答項目のみ追加生成する。**

回答ファイルが部分的に存在する状態は実運用で起こりやすい。これを毎回全再生成にすると、既存の判断や文章が上書きされ、履歴追跡もしにくくなる。
そのため、回答ファイルが部分的に存在する場合は、既存回答を保持し、未回答の質問だけを追加生成する方式とする。
ただし、質問文自体が変更された場合や回答整合性が崩れた場合は、対象回答ファイル全体を再生成してよい。

### Q4. `IMPLEMENTATION_PLAN.md` を承認対象から省略できる条件（セクション16）

「必要に応じて承認対象に含める」とあるが、含めない場合の判断基準が不明。
どのような場合に省略可能か。

**A4. 実装方針が SPEC.md・CLASS.md・TEST.md に十分具体化されており、別途実装計画書を作る必要がない小規模案件では省略可能とする。**

省略可能なのは以下のようなケースである。

- 小規模または単機能である
- 実装単位が明確で、タスク分解が不要
- 実装順序や分割方針が設計書から十分読み取れる
- 外部連携や移行作業など、追加の実装計画が不要

逆に、複数モジュール・複数画面・複数API・移行手順・段階リリースなどがある場合は、`IMPLEMENTATION_PLAN.md` を承認対象に含める。

### Q5. `review_requested` 状態の遷移タイミング（セクション7・13）

状態定義に `review_requested` が存在するが、セクション13の自動進行ループにはこの状態への遷移が記述されていない。
`review_requested` はどのタイミングで付与され、誰が付与するか。

**A5. `review_requested` は、成果物生成または改訂が完了し、レビュー担当AIへレビュー依頼を出した時点で Claude Code が付与する。**

自然な状態遷移は次の通り。

- 新規作成直後: `drafted`
- Claude Code がレビュー依頼を発行: `review_requested`
- レビュー報告生成後: `reviewed`
- 質問あり: `questions_pending`
- 回答反映後: `revised`
- 再レビュー依頼時: 再度 `review_requested`
- 承認時: `approved`

したがって、`review_requested` は Claude Code が外部レビュー処理を起動した時点で設定する状態と定義する。

### Q6. `state.json` テンプレートの `project_name` の置換方法（セクション15）

テンプレートの `project_name` が `"sample-project"` でハードコードされている。
実際の利用時にどのように置換するか（ルートフォルダ名を自動取得するか、手動設定か）。

**A6. 既定ではルートフォルダ名を自動取得して設定し、取得できない場合のみ `"sample-project"` をフォールバック値として使う。**

セクション14.2 では `project_name` は「ルートフォルダ名、または固定名」とされている一方、テンプレートでは `"sample-project"` が書かれている。ブートストラップ時にルートディレクトリ名を自動取得して `project_name` に設定することを標準とし、何らかの理由で取得できない場合のみ `"sample-project"` を使う。
これでテンプレートは雛形として残しつつ、実運用では自動置換できる。
