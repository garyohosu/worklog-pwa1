## 指摘事項

### 1. [重大] admin の所有権仕様が矛盾

admin は他ユーザー記録も「作成」可能と読める一方で、`action=create` は `user_id` を受け取らず常に Bearer Token のユーザーへ紐付けるため、代理作成の可否と方式が確定していない。

- SPEC.md:129, 137, 557, 673

### 2. [重大] セッション期限が固定7日とスライディング7日で衝突

`expires_at` とログイン成功時の説明は固定期限だが、セキュリティ要件では最終操作から延長されるスライディング方式。期限判定と更新処理を実装できない。

- SPEC.md:267, 302, 324, 845

### 3. [重大] 削除/状態変更 API が `revision` 競合制御を満たしていない

同期仕様では更新・削除で `base_revision` 必須だが、`DELETE` と `action=status` の入力仕様にその受け渡し方法がない。

- SPEC.md:511, 560, 677, 679, 707

### 4. [高] `work_logs` が画面要件の監査時刻を保持できない

詳細画面は「作成日時 / 最終更新日時 / 同期日時」を要求しているが、テーブルには `created_at` / `updated_at` がなく、`recorded_at` の形式・タイムゾーンも未定義。

- SPEC.md:212, 215, 220, 422

### 5. [高] 一覧画面の検索条件に対する API 契約が不足

UI では設備名・設備コード・ライン・日付・種別・状態・削除済み条件を持つのに、API 側は `page` と `page_size` しか定義されていない。

- SPEC.md:392, 399, 669, 701

### 6. [中] 仮パスワード再設定後の強制変更フローが未完

「次回ログイン後に変更を促す」とある一方、その状態を保持する項目やログイン時の返却仕様がない。

- SPEC.md:147, 152, 165, 688

### 7. [中] 機能の優先度分類が一貫していない

機能要件ではカメラ利用が必須だが、写真は MVP 外で、MVP 完成条件にもカメラ/写真は入らない。必須機能と MVP 範囲を再定義すべき。

- SPEC.md:93, 625, 988

### 8. [中] CORS 許可オリジンが開発環境で曖昧

Origin はポート込みで判定されるが、列挙値にポートがなく、ローカル開発サーバーの許可条件が定まらない。

- SPEC.md:112, 847
